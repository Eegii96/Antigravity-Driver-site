## 4. Хэрэглэгчийн Үүрэг ба Бүртгэлийн Логик

### А. Хэрэглэгчийн хоёр үүрэг
| Код | Харагдах нэр |
|---|---|
| `operator` | Жолооч / Оператор |
| `employer` | Ажил олгогч / Захиалагч |

### Б. Бүртгэлийн нэршлийн дүрэм
- "Ажил олгогч / Захиалагч" үүрэг сонгоход нэр бөглөх талбарыг **"Овог нэр / Компанийн нэр"** болгоно.

### В. Үүрэг шилжих үеийн алдаагүй байдал
- Бүртгэлийн явцад үүрэг (`operator` ⇄ `employer`) хооронд шилжих үед намтар үүсгэх бүх төлөвийг бүрэн цэвэрлэнэ:
  - `setBio('')`, `setOriginalBio('')`, `setHasOptimized(false)`
- Шинэ үүргийн дагуу AI намтар шинээр үүсдэг байна (өмнөх намтар хэвээр үлдэж болохгүй).

## 5. AI Намтар Үүсгэх Дүрэм

Бид `gemini-2.5-flash` моделиор хэрэглэгчийн намтрыг (bio) сайжруулдаг.

### А. Жолооч / Операторын намтар
- **Заавал нэгдүгээр биеэр бичнэ** — `"Би..."`, `"Миний бие..."`, `"Миний ажилласан хугацаанд..."` гэх мэт.
- Гуравдагч биеэр (`"Дорж нь..."`, `"Тэрээр..."`) хэзээ ч бичиж болохгүй.
- **Онцлох агуулга:** сахилга бат, архинаас татгалзсан байдал, ажлын хариуцлага, техникийн өдөр тутмын арчилгаа.

### Б. Ажил олгогч / Захиалагчийн намтар
- **Хувь хүн** (нэрэнд ХХК, Компани гэх мэт байхгүй) → нэгдүгээр биеэр, хувь хүний зүгээс: `"Би ажил олгогч Дорж байна..."` — "манай компани" гэж хэзээ ч бичихгүй.
- **Байгууллага / Компани** (нэрэнд ХХК, Компани гэж орсон) → компанийн зүгээс: `"Манай компанийн зүгээс..."`, `"Бид хамт олондоо..."`.
- **Онцлох агуулга:** цалингийн найдвартай байдал, HSE аюулгүй байдлын соёл, кэмп болон хоол хүнсний хангалт.

### В. Техникийн шаардлагууд
- Намтар сайжруулах явцад хэрэглэгчид **"AI сайжруулж байна..."** loading state заавал харуулна.
- AI-ийн хариултаас markdown тэмдэглэгээ (`**`, `*`) автоматаар цэвэрлэж, цэвэр энгийн текст буцаана.

## 7. Next.js Хөгжүүлэлтийн Дүрмүүд

> [!IMPORTANT]
> **Дүрэм 1 — Client vs Server Components**
> Next.js-д файл бүр анхнаасаа Server Component байдаг. `useState`, `useEffect`, `useRouter` зэрэг React hook эсвэл `window`, `localStorage`, `document` ашиглах бол файлын **хамгийн эхний мөрөнд** заавал `'use client';` гэж бичнэ. Байхгүй бол build алдаа заана.
>
> **Дүрэм 2 — SEO Metadata & Client/Server зааглалт**
> `'use client'` заавартай хуудаснаас `metadata` экспортлох боломжгүй. Иймд:
> - `page.tsx` → Server Component байлгаж, `export const metadata` эсвэл `generateMetadata` тодорхойлно.
> - React hook ашиглах логикийг тусдаа `*Client.tsx` файл болгон салгаж, `page.tsx`-ээс дуудна (жишээ нь: `AuthClient.tsx`, `BoardClient.tsx`).
>
> **Дүрэм 3 — Орчны хувьсагч (.env)**
> Хэрэглэгчийн хөтөч дээр уншигдах хувьсагчид заавал `NEXT_PUBLIC_` угтвартай байна (`NEXT_PUBLIC_FIREBASE_API_KEY`, `NEXT_PUBLIC_GEMINI_API_KEY` г.м). Серверт зориулсан нууц хувьсагчид угтваргүй байна.
>
> **Дүрэм 4 — Static Export & generateStaticParams**
> `next.config.ts` дотор `output: "export"` тохиргоотой тул dynamic routes (`/jobs/[id]`, `/profile/[id]`) нь `generateStaticParams()` функцаар build хийх үед бүх ID-г Firestore-оос уншиж pre-render хийнэ. Сүлжээний алдаанаас сэргийлж seed mock ID-нуудыг **үргэлж fallback** болгон бэлдсэн байна.

### А. App Router-ийн хуудас бүтэц (одоогийн байдал)

| URL | Файл | Тайлбар |
|---|---|---|
| `/` | `src/app/page.tsx` | Нэвтрэлт шалган `/board` эсвэл `/auth` руу шилжүүлнэ |
| `/auth` | `src/app/auth/page.tsx` + `AuthClient.tsx` | Нэвтрэх / бүртгүүлэх |
| `/board` | `src/app/board/page.tsx` + `BoardClient.tsx` | Ажлын зарын самбар |
| `/settings` | `src/app/settings/page.tsx` + `SettingsClient.tsx` | Тохиргоо |
| `/profile` | `src/app/profile/page.tsx` | Өөрийн профайл руу шилжүүлнэ |
| `/profile/[id]` | `src/app/profile/[id]/page.tsx` + `ProfileDetailClient.tsx` | Динамик профайл (SEO) |
| `/applications` | `src/app/applications/page.tsx` + `ApplicationsClient.tsx` | Миний хүсэлтүүд |
| `/jobs/[id]` | `src/app/jobs/[id]/page.tsx` + `JobDetailClient.tsx` | Динамик зарын хуудас (SEO) |

### Б. SEO Metadata-ийн Монгол Дүрэм

Динамик зарын гарчиг үүсгэхэд дараах хэлбэржилтийг баримтална:

- **Байршлын залгавар:**
  - `... аймаг` → `... аймагт ажиллах`
  - `... дүүрэг` → `... дүүрэгт ажиллах`
  - `... сум` → `... суманд ажиллах`
  - `... хот` → `... хотод ажиллах`
- **Цалингийн хэлбэржилт:**
  - 1,000,000+ → `"3.5 сая"`
  - 1,000+ → `"120 мянга"`
- **Зарын гарчгийн загвар:**
  `[Байршил] ажиллах [Техникчин] яаралтай авна — Цалин [Дүн] ([Нэгж]) | Жолооч Монголиа`
- **Профайлын гарчгийн загвар (Оператор):**
  `[Нэр] — Хүнд механизмын оператор (Туршлага: X жил, Үнэлгээ: X★) | Жолооч Монголиа`
- **Профайлын гарчгийн загвар (Ажил олгогч):**
  `[Нэр] — Ажил олгогч ([Компанийн нэр]) | Жолооч Монголиа`

## 8. Firebase & GitHub Удирдлага

### А. Build & Deploy ажлын урсгал

Кодонд өөрчлөлт оруулах бүрд дараах дарааллыг чанд баримтална:

1. **`npm run build`** — TypeScript болон бүтцийн алдааг шалгана. Алдаатай бол deploy хийж болохгүй.
2. **`npm run deploy:firebase`** — Build амжилттай болсны дараа шинэчлэлийг амьд сайт руу илгээнэ.
3. **`git commit` & `git push`** — Бүх өөрчлөлтийг ойлгомжтой англи тайлбартай commit хийж `main` руу push хийнэ.
4. **`npm run dev`** — Зөвхөн локал орчинд туршихад ашиглана (deploy хийхийн өмнө биш).

> [!WARNING]
> Browser tool / Chrome DevTools subagent ашиглан нэмэлт туршилт хийхийг **хориглоно** — маш их цаг алдуулдаг тул хийх шаардлагагүй. Build & deploy амжилттай болсон бол шууд хэрэглэгчид танилцуулна.

### Б. Git & Аюулгүй байдал
- `.env` файл, `.firebase/` хавтас, `*.zip` нөөц файлуудыг **хэзээ ч GitHub руу push хийж болохгүй** (`.gitignore`-д бүрэн хаасан).
- Firebase Firestore болон Storage-ийн Security Rules нь зөвхөн бүртгэлтэй хэрэглэгч өөрийн өгөгдлийг унших/бичих зөвшөөрөлтэй байна.

### В. Firebase Hosting тохиргоо
- `firebase.json` дотор `"public": "out"` тохируулсан.
- `rewrites` тохиргоогоор бүх хүсэлтийг `index.html` руу чиглүүлж 404 алдаанаас сэргийлнэ.

## 9. Ажлын Журам (AI Агентын Дүрэм)

### А. Төлөвлөгөө гаргах журам
- **Жижиг ажил** (алдаа засах, нэг файл өөрчлөх, жижиг UI засвар) → Шууд хийж танилцуулна, төлөвлөгөө гаргах шаардлагагүй.
- **Том өөрчлөлт** (шинэ том модуль, архитектурын шийдвэр, олон файл өөрчлөх) → `implementation_plan.md` үүсгэж хэрэглэгчээр **заавал батлуулсны дараа** хийнэ.

### Б. Агентын чадвар ба журам
- Агент нь хэрэглэгчийн зөвшөөрлийн дагуу файлуудыг шууд өөрчлөх, терминал команд ажиллуулах чадвартай (хэрэглэгч гараар хуулж тавих шаардлагагүй).
- Алдааг Chrome DevTools болон консол лог дэмжлэгтэйгээр бие даан илрүүлж засна.

### В. Subagent & Параллел хөгжүүлэлт
- Томоохон бүтцийн өөрчлөлт, олон хамааралгүй файлуудыг зэрэг засах шаардлагатай үед агент **өөрөө шийдвэр гарган subagent-уудыг зэрэгцүүлэн (parallel) ажиллуулна.**
- Агент бүрд өөр өөр файл/модуль хариуцуулж, кодын зөрчил (merge conflict) гарахаас сэргийлнэ.
- Subagent-уудын ажил дуусмагц үндсэн агент нэгтгэж, `npm run build` хийсний дараа Firebase руу deploy хийнэ.

### Г. Шинэ харилцан яриа эхлүүлэх үед
- Хэрэглэгч **"Дүрмээ хар"** эсвэл **"Төслийн дүрмээ унш"** гэж хэлэх бүрд AI агент энэхүү [design_guide.md](file:///c:/Энхчулуун/1. Сайт/Antigravity Driver site/design_guide.md) файлыг уншиж, бүх дүрэм, технологийн стек, хуудас бүтэц, SEO логик, subagent журам болон deploy урсгалыг **бүрэн сэргээн санаж** шууд ажилд бэлэн байна.
